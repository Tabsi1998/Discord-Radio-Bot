# OmniFM Website Foundation

## Purpose

This document is the Phase 0 and Phase 1 contract for the public OmniFM website.
It defines:

- who the website is for
- what each tier must communicate
- which strengths must be visible above the fold and which belong deeper on the page
- how the homepage should guide a new visitor from understanding to invite or upgrade

This contract exists so product, frontend, and marketing stay aligned.

## Core Question

The website must answer these questions in order:

1. What is OmniFM?
2. Why is OmniFM better than a basic Discord radio bot?
3. How quickly can I start?
4. Why would I upgrade from Free to Pro or Ultimate?

## Primary Audiences

### Free

- small Discord communities
- private friend groups
- users who want instant radio with no setup friction

### Pro

- community admins
- event and activity servers
- teams that need scheduling, dashboard control, and role permissions

### Ultimate

- operators and power users
- larger communities
- users who need maximum reliability, custom stations, analytics, and automation

## Product Positioning

### OmniFM

OmniFM is a Discord radio platform with 24/7 streaming, a commander and worker architecture, live station access, and structured upgrade paths for server operators.

### Free promise

Start fast, stream reliably, and get real radio on your Discord server without friction.

### Pro promise

Operate your server professionally with dashboard control, events, permissions, and better oversight.

### Ultimate promise

Run OmniFM like infrastructure with custom station control, stronger reliability tooling, analytics, and automation hooks.

## Messaging Rules

- Lead with user benefit before architecture.
- Show proof before making larger plan claims.
- Do not market a capability that is not implemented.
- Do not bury Pro and Ultimate value under generic feature lists.
- Explain technical terms in plain language.
- Keep German and English content functionally identical.

## Homepage Job Per Section

### Hero

Job:
Explain OmniFM in one sentence and push to the primary action.

Must communicate:

- Discord radio
- 24/7 availability
- large station catalog
- better reliability or control than a basic bot

Primary CTA:

- invite the bot

Secondary CTA:

- understand how it works

### Trust Bar

Job:
Provide immediate proof with compact, credible numbers and operational signals.

Preferred proof points:

- live stations
- active bots or workers
- supported plans
- dashboard availability

### How It Works

Job:
Reduce setup anxiety and make the first-run flow feel easy.

Flow:

1. Invite bot
2. Run `/play`
3. Listen or scale with workers and plans

### Why OmniFM

Job:
Translate product strengths into user outcomes.

Must cover:

- 24/7 streaming
- slash command simplicity
- worker-based scaling
- reconnect and reliability
- dashboard control
- premium growth path

### Dashboard Showcase

Job:
Make Pro and Ultimate feel real, not abstract.

Must highlight:

- event scheduler
- role permissions
- health and analytics
- custom stations or exports for Ultimate

### Reliability Section

Job:
Explain why the architecture matters without leading with jargon.

Translate these terms:

- workers
- reconnect
- failover
- health

### Use Cases

Job:
Explain which audience and server type matches Free, Pro, and Ultimate.

Must answer:

- who should stay on Free
- who should upgrade to Pro
- who needs Ultimate
- why the tiers are different in practice

### Pricing

Job:
Make the upgrade path obvious and fair.

Must answer:

- what Free includes
- why Pro is worth it
- why Ultimate exists

### Commands

Job:
Support evaluation, not lead the sale.

Commands belong after value and pricing, not before.

### FAQ

Job:
Resolve hesitation before the visitor leaves.

Must cover:

- how to start
- what is free
- when Pro is needed
- when Ultimate is needed
- how workers work

## Content Priority

### Must be visible on the homepage

- 24/7 radio for Discord
- station count
- direct invite path
- worker and reliability advantage in plain language
- dashboard value
- clear Free, Pro, Ultimate difference

### Can live lower on the homepage

- full command listings
- deep operational details
- long capability matrices

### Keep out of the first impression

- raw implementation details
- internal naming without explanation
- weak metrics that confuse trust

## Definition Of Done

- A new visitor understands OmniFM within 5 seconds.
- The homepage explains why OmniFM is stronger than a basic radio bot.
- Free, Pro, and Ultimate have distinct identities.
- The invite path is direct and obvious.
- German and English stay aligned.
- Homepage structure reflects value first, implementation second.
